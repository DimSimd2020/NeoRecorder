# Архитектура NeoRecorder

Этот документ определяет архитектурные правила проекта.

Все разработчики и AI-агенты обязаны следовать этим правилам.
Нарушение архитектурных границ приводит к техническому долгу и нестабильности системы.

Основные принципы:

- строгая изоляция доменной логики
- явные границы между слоями
- предсказуемая модель состояний
- минимальная связанность модулей
- наблюдаемость ошибок
- отсутствие скрытых побочных эффектов

---

# Общая модель системы

NeoRecorder — это модульная система записи и студийного управления сценами.

Архитектура построена вокруг нескольких изолированных доменных областей (bounded contexts) и слоёв.

---

# Доменные области (Bounded Contexts)

## Studio

Отвечает за студийную модель сцены.

Функции:

- сцены
- источники
- композиция сцены
- preview/program режим
- переходы между сценами
- граф сцены

Примеры:

core/studio/

---

## Capture

Отвечает за получение медиа с устройств и операционной системы.

Функции:

- захват экрана
- захват окна
- захват региона
- микрофон
- системный звук
- обнаружение устройств

Примеры:

utils/display_manager  
utils/audio_manager  
utils/screenshot

---

## Recording / Output

Отвечает за запись и кодирование потоков.

Функции:

- управление процессом записи
- выбор кодеков
- оркестрация ffmpeg
- стриминг
- сохранение файлов
- буфер повторов (replay buffer)

Примеры:

utils/ffmpeg_handler  
core/recorder
core/runtime

---

## Media

Отвечает за управление записанными медиа-ресурсами.

Функции:

- управление записанными файлами
- метаданные
- превью и миниатюры
- извлечение клипов
- хранение данных о медиа

---

## AI / Analysis

Отвечает за анализ записанного контента.

Функции:

- транскрипция
- определение хайлайтов
- генерация субтитров
- анализ контента

ВАЖНО:

AI-модуль **не должен влиять на realtime-pipeline записи**.

Он работает только **асинхронно после записи**.

---

# Слои системы

## Domain

Доменная логика системы.

Содержит:

- модели
- бизнес-правила
- состояние студии
- управление сценами

Доменный слой:

- не знает о GUI
- не знает о ffmpeg
- не знает о внешних библиотеках захвата

---

## Application

Оркестрация доменных операций.

Содержит:

- сервисы
- сценарии использования
- координацию действий

---

## Infrastructure

Работа с внешними системами.

Содержит:

- ffmpeg
- устройства
- файловую систему
- операционную систему
- сетевые операции

---

## UI

Графический интерфейс пользователя.

Содержит:

- окна
- панели
- визуализацию сцены
- элементы управления

UI **никогда не содержит бизнес-логики**.

---

# Разрешённые зависимости

Разрешённая схема зависимостей:

UI → Application → Domain  
Infrastructure → Domain  
Application → Infrastructure

Запрещено:

Domain → UI  
Domain → Infrastructure  
UI → Infrastructure напрямую

---

# Что считается доменной логикой

К доменной логике относятся:

- модели сцен
- источники
- состояние студии
- transitions
- правила композиции
- правила микширования

Эта логика должна находиться **только в domain/core слоях**.

---

# Что считается инфраструктурой

Инфраструктура — это взаимодействие с внешним миром.

К инфраструктуре относятся:

- ffmpeg
- устройства захвата
- микрофоны
- системные API
- файловая система
- сетевые протоколы
- библиотеки захвата

---

# Где запрещены shortcuts

Запрещено:

- вызывать ffmpeg напрямую из UI
- изменять состояние сцены из GUI
- хранить состояние системы в UI-виджетах
- обходить сервисный слой
- напрямую модифицировать глобальное состояние

Любое изменение состояния должно проходить через **Application или Domain сервисы**.

---

# Модель состояния

Все важные подсистемы должны иметь явную модель состояния.

Например:

- Recorder state machine
- Studio session state
- Broadcast runtime state
- Output session state
- Scene state
- Source state

Запрещены:

- неявные переходы состояния
- изменение состояния без логирования
- скрытые side-effects

---

# Логирование

Любая ошибка должна:

- быть залогирована
- иметь контекст
- не ломать pipeline без сообщения

Silent failures запрещены.
